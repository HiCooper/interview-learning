## 一、为什么上下文管理是核心挑战

### 三个硬约束

| 约束 | 说明 |
|------|------|
| **窗口上限** | LLM 上下文有硬限制（128K–1M tokens） |
| **成本** | API 按 token 计费，长上下文 = 高成本 |
| **注意力衰减** | 模型对上下文中间部分关注度下降（Lost in the Middle） |

### Agent 面临的特殊挑战

- **多轮对话**：历史不断增长
- **多步任务**：每步中间结果占用上下文
- **工具调用**：工具定义 + 返回值占据大量 token
- **长任务**：复杂任务可能跨数小时甚至数天

> 上下文管理不是"有没有空间"的问题，而是**在有限资源内最大化任务成功率**的工程问题。

---

## 速查卡

- **三个硬约束**：窗口上限（128K-1M tokens）→ 成本（按 token 计费）→ 注意力衰减（Lost in the Middle）
- **基础策略**：滑动窗口（截断最早内容）→ 摘要压缩（生成摘要替代原文）→ 选择性保留（向量检索相关片段）
- **Claude Code 分层**：系统级（Prompt+CLAUDE.md+Memory）→ 会话级（对话+工具调用+Todo）→ 按需加载（Git+文件+搜索）
- **Memory 系统**：文件式持久记忆，索引+详情分离，Frontmatter 元数据，[[wiki-link]] 关联，增量更新
- **上下文压缩**：早期对话→结构化摘要（保留任务目标/关键决策/代码变更/TODO），透明注入 system-reminder
- **子 Agent 隔离**：独立上下文+Worktree 隔离，结构化返回结果，主 Agent 只看结论不看过程
- **实践建议**：约定文件（项目知识）→ 文件式 Memory（用户偏好）→ 早期压缩+近期保留（对话管理）→ Todo+Checkpoint（长任务）

## 二、基础策略

### 滑动窗口

只保留最近 N 轮，丢弃更早内容。简单但丢失早期关键信息。

### 摘要压缩

对早期对话生成摘要替代原始对话。挑战：摘要可能遗漏细节。

### 选择性保留

只保留与当前问题相关的历史。通常结合向量检索：历史消息 embedding → 检索相关片段 → 拼入上下文。

## 三、Claude Code 上下文管理（深度解析）

Claude Code 是目前 Agentic Coding 工具的标杆。

### 整体架构

```
┌─────────────────────────────────────────────┐
│              系统级上下文（持久）              │
│  系统 Prompt + CLAUDE.md + Memory 文件        │
├─────────────────────────────────────────────┤
│              会话级上下文（动态）              │
│  当前任务 + 对话历史 + 工具调用结果 + Todo     │
├─────────────────────────────────────────────┤
│              按需加载（即时）                 │
│  Git 状态 + 文件内容 + 搜索结果                │
└─────────────────────────────────────────────┘
```

### 1. CLAUDE.md — 项目级持久上下文

放在项目根目录的约定文件，**每次会话启动时自动加载**。

```markdown
# CLAUDE.md

## 技术栈
Java 17, Spring Boot 3.2, PostgreSQL, JPA

## 代码规范
- Controller 只做参数校验和路由
- 业务逻辑在 Service 层
- 异常统一用 @ControllerAdvice

## 命名约定
Service: XxxService, DTO: XxxRequest/Response
```

**设计精髓**：
- **约定优于配置**：纯文本文件，Git 可追踪
- **作用域清晰**：根目录作用于全局，子目录可覆盖
- **人在回路**：开发者维护，Agent 消费
- **Token 高效**：只在会话开始加载一次

### 2. Memory 系统 — 跨会话持久记忆

文件式 Memory 是 Claude Code 最独特的设计之一。

**存储结构**：

```
~/.claude/projects/<project>/memory/
├── MEMORY.md               # 索引（启动时加载）
├── user-preference.md      # "函数式风格偏好"
└── login-bug-fix.md        # "登录超时的根因和修复"
```

**Memory 文件格式**：

```yaml
---
name: user-preference
description: 用户偏好函数式风格
metadata:
  type: user
---

用户偏好函数式编程，优先使用不可变数据结构。

**Why:** 代码可预测性更高
**How to apply:** 新功能默认函数式风格
```

**MEMORY.md 索引**：

```markdown
- [User preference](user-preference.md) — 函数式风格
- [Login bug fix](login-bug-fix.md) — 超时根因
```

**设计精髓**：
- **索引 + 详情分离**：MEMORY.md 一行概括，详细内容在独立文件
- **Frontmatter 元数据**：类型标记（user/feedback/project），便于过滤
- **关联链接**：`[[other-memory]]` 建立记忆间关联图
- **增量更新**：一个记忆一个文件，增删方便

### 3. 上下文压缩（Compaction）

对话接近窗口上限时自动触发：

```
压缩前: [N 轮完整对话]
    ↓
压缩: 早期对话 → 结构化摘要
      保留: 任务目标、关键决策、代码变更、TODO 状态
    ↓
压缩后: [结构化摘要] + [最近几轮完整对话]
```

**关键设计**：
- **透明**：压缩后以 `<system-reminder>` 注入，Agent 知道被压缩了
- **结构化**：保留的是结构化信息而非对话原文
- **渐进**：只压缩早期对话，最近交互保持完整

### 4. 子 Agent 上下文隔离

主 Agent 将子任务委托给独立 Agent，各自拥有独立上下文：

```
主会话（用户任务 + 全局上下文）
    ├── Agent A: 搜索代码库（上下文 = 搜索结果）
    ├── Agent B: 分析架构（上下文 = 架构文件）
    └── Agent C: 实现功能（上下文 = 相关代码 + Worktree 隔离）
```

- **Worktree 隔离**：子 Agent 在独立 Git worktree 中工作
- **结构化返回**：返回结果（text 或 JSON Schema），不返回完整对话
- **Token 节省**：主 Agent 只看结论，不看中间过程

### 5. Todo 系统

复杂任务创建 Todo 列表，跨轮持久化：

```json
[
  {"id":"1", "subject":"创建数据模型", "status":"completed"},
  {"id":"2", "subject":"实现Service", "status":"in_progress"},
  {"id":"3", "subject":"写测试", "status":"pending", "blockedBy":["2"]}
]
```

支持依赖关系（blocks/blockedBy）和状态机（pending→in_progress→completed）。

### 6. 设计原则总结

| 原则 | Claude Code 做法 |
|------|-----------------|
| **分层** | 系统级 → 会话级 → 按需加载 |
| **持久化关键信息** | CLAUDE.md + Memory + Todo |
| **压缩而非丢弃** | 结构化摘要替代截断 |
| **隔离上下文** | 子 Agent 独立上下文 |
| **用户可控** | 开发者维护持久配置文件 |
| **透明可见** | 每次状态变更都有 system-reminder |

## 四、其他业界方案

### Cursor Rules

`.cursor/rules/` 目录，支持 glob 模式匹配不同文件类型应用不同规则。更细粒度但配置量更大。

### LangChain Memory 体系

| 类型 | 说明 |
|------|------|
| `ConversationBufferMemory` | 存储全部对话 |
| `ConversationSummaryMemory` | LLM 摘要压缩 |
| `ConversationBufferWindowMemory` | 滑动窗口 |
| `VectorStoreRetrieverMemory` | 向量检索相关记忆 |

> 灵活但需开发者手动选择和组合，缺乏开箱即用的最佳实践。

### GitHub Copilot

隐式上下文收集：当前文件、相邻标签页、Git 历史、语言服务器。减少了配置但可控性不如显式的 CLAUDE.md。

## 五、实践建议

### 构建 Agent 上下文系统

| 层级 | 建议 |
|------|------|
| 项目知识 | 类似 CLAUDE.md 的约定文件 |
| 用户偏好 | 文件式 Memory，跨会话持久化 |
| 对话管理 | 早期压缩 + 近期保留 |
| 长任务 | Todo + checkpoint 机制 |
| 并行工作 | 子 Agent 独立上下文 |
| 成本控制 | 设定每轮 token 预算 |

### 常见陷阱

| 陷阱 | 说明 |
|------|------|
| 塞进所有信息 | Lost in the Middle，放进去 ≠ 被用到 |
| 简单截断 | 丢失早期关键信息 |
| 不记录决策 | 多轮后忘记之前的选择 |
| 忽略成本 | 长上下文显著增加延迟和费用 |
| 过度依赖 Agent | 关键记忆应由人维护 |

## 自测

1. **为什么"简单截断"是最差的上下文管理策略？**
   <br/>→ 截断丢弃早期信息可能包括任务目标、关键决策和约束条件——丢失后 Agent 可能偏离方向或重复已解决的问题。"压缩而非丢弃"（结构化摘要替代原文）是更好的选择。

2. **Claude Code 的子 Agent 上下文隔离为什么能节省 Token？**
   <br/>→ 子 Agent 独立上下文只包含任务相关的代码/文件，主 Agent 只接收结构化返回结果（text 或 JSON），不接收完整对话过程。类似"只汇报结论，不汇报过程"，大幅减少主会话的 token 消耗。

3. **文件式 Memory（Claude Code）和向量存储 Memory（LangChain）各有什么适用场景？**
   <br/>→ 文件式适合结构化知识（用户偏好、项目规范、关键决策），人可读可维护，跨会话精确加载。向量存储适合海量非结构化记忆（对话历史、文档库），支持语义检索但精确度不如文件式。两者互补：文件式存储"必须记住的"，向量存储处理"可能需要参考的"。
