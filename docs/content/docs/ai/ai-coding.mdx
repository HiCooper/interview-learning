## 一、范式转变

```
传统：思考 → 写代码 → 调试 → 测试
AI：  思考 → 描述要求 → AI 生成 → 审查 → 调试 → 测试
```

**核心能力转移**：编码能力 → 需求描述能力 + 代码审查能力

### AI 编程工具谱系

| 类型 | 代表 | 能力 |
|------|------|------|
| **代码补全** | GitHub Copilot, Cursor Tab | 行级/块级补全 |
| **对话式** | ChatGPT, Claude Chat | 问答、生成、解释 |
| **Agentic** | Claude Code, Cursor Agent | 读写文件、执行命令、Git 操作 |
| **全自主** | Devin, Factory | 端到端开发任务 |

## 速查卡

- **范式转变**：编码能力 → 需求描述+代码审查能力。AI 是副驾驶，不是自动驾驶
- **AI 工具谱系**：代码补全（Copilot）→ 对话式（ChatGPT）→ Agentic（Claude Code/Cursor Agent，读写文件+执行命令+Git）→ 全自主（Devin）
- **Prompt 技巧**：结构化描述（技术栈+接口+约束）→ Spec-driven → 迭代精炼 → 示例驱动
- **Agentic Coding**：读取代码库→理解上下文→修改文件→运行测试→修复错误→Git 操作，完整闭环
- **审查要点**：正确性+安全性+性能+可维护性+依赖。AI 代码常见问题：幻觉 API、过度实现、忽略异常、不安全默认
- **工作流**：新功能=Spec→AI实现→Review→测试→PR；Bug=描述+复现→AI定位→修复方案→Review→回归测试
- **CLAUDE.md**：项目规范文件，技术栈+代码规范+命名约定，Git 可追踪，每次会话自动加载

## 二、Prompt 工程（代码场景）

### 结构化描述

```
差的: "帮我写个登录"
好的:
"用 Spring Boot 实现登录 API：
- POST /api/login
- 接收 { username, password }
- 返回 JWT token（24h 有效期）
- 密码 BCrypt 加密
- 失败返回 401 + 错误原因
- IP 限流：每分钟最多 5 次"
```

### Spec-driven 开发

先写详细 Spec（接口定义、请求/响应格式、约束条件） → AI 实现 → 根据 Spec 验证。

### 迭代式精炼

生成初始版本 → Review 指出问题 → AI 修改 → 多轮迭代 → 最终审查和测试。

### 示例驱动

```java
// 参考此风格实现 updateUser：
public UserResponse createUser(CreateUserRequest req) {
    validateRequest(req);
    User user = userMapper.toEntity(req);
    user = userRepository.save(user);
    return userMapper.toResponse(user);
}
```

## 三、Agentic Coding

> Agentic Coding 工具自主：读取代码库 → 理解上下文 → 修改文件 → 运行测试 → 修复错误 → Git 操作，形成完整开发闭环。

### Claude Code 工作模式

```
用户指令 → 读取代码库 → 制定计划
  → 编辑文件 → 运行测试 → 修复 → 提交
  → 失败自动分析 → 修复 → 重测
```

### 高效使用技巧

| 技巧 | 说明 |
|------|------|
| **明确上下文** | 告知技术栈和约束 |
| **先探索后修改** | 先让 Agent 了解代码库 |
| **分步验证** | 每完成子任务后运行测试 |
| **详细 Spec** | 需求越具体，结果越可靠 |
| **审查 diff** | 像 Code Review 一样审查 |
| **项目规范** | 用 CLAUDE.md 设定规则 |

### CLAUDE.md / Cursor Rules

```markdown
# 项目技术栈
Java 17, Spring Boot 3.2, PostgreSQL, JPA

# 代码规范
- Controller 只做参数校验和路由
- 业务逻辑在 Service 层
- 统一异常处理用 @ControllerAdvice
- 测试用 JUnit 5 + MockMvc

# 命名约定
Service: XxxService, DTO: XxxRequest/Response
```

## 四、AI 代码审查要点

| 维度 | 关注点 |
|------|--------|
| **正确性** | 逻辑正确？边界条件？ |
| **安全性** | SQL 注入？XSS？敏感数据泄露？ |
| **性能** | N+1 查询？大循环重复计算？ |
| **可维护性** | 命名清晰？职责单一？ |
| **依赖** | 是否引入不合适库？版本兼容？ |

### AI 代码常见问题

- **幻觉 API**：调用不存在的库方法
- **过度实现**：需求简单但架构复杂
- **忽略异常**：错误处理不完整
- **不安全默认**：未开启安全配置
- **风格不一致**：与现有代码不匹配

## 五、工作流最佳实践

### 新功能开发

```
写 Spec（人工） → AI 生成实现 → Review diff → AI 生成测试 → 运行 → 修复 → PR
```

### Bug 修复

```
描述 Bug + 复现步骤 → AI 定位 → AI 修复方案 → Review → 确认 → AI 写回归测试
```

### 代码重构

```
描述目标 → AI 分析影响 → 分步执行 → 每步验证 → Review 最终 diff
```

### 关键原则

> **AI 是副驾驶，不是自动驾驶。** 最终决策权和责任在开发者。

## 自测

1. **AI 生成代码的常见问题有哪些？审查时需要重点关注什么？**
   <br/>→ 幻觉 API（调用不存在的库方法）、过度实现（需求简单但架构复杂）、忽略异常处理、不安全默认配置、风格与现有代码不匹配。审查重点：正确性→安全性（注入/泄露）→性能（N+1 查询）→可维护性→依赖合理性。

2. **CLAUDE.md 的设计精髓是什么？相比 Cursor Rules 有什么优劣？**
   <br/>→ 精髓：约定优于配置、作用域清晰（根目录全局+子目录覆盖）、人在回路（开发者维护）、Token 高效（只加载一次）。vs Cursor Rules：CLAUDE.md 更简单（一个文件），Cursor Rules 更细粒度（glob 模式匹配不同文件类型），但配置量大。

3. **Agentic Coding 与代码补全工具的本质区别是什么？**
   <br/>→ 代码补全是被动响应（你打字它补全），Agentic Coding 是主动执行（读取代码库→理解上下文→修改多文件→运行测试→修复→提交），形成完整的自主开发闭环。前者辅助编码，后者自主完成任务。
