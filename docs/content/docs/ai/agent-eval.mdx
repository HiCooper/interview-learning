## 一、为什么 Agent 评测难？

- **多步推理**：单轮正确不代表整个任务完成
- **工具调用**：需评估选对了工具和参数
- **部分正确**：可能部分正确但非最优
- **非确定性**：相同输入可能不同输出

## 速查卡

- **Agent 评测难点**：多步推理→工具调用→部分正确→非确定性
- **评测维度**：任务完成率→工具调用正确率→效率（步骤/延迟）→回复质量→安全通过率→鲁棒性（错误恢复）
- **LLM-as-Judge**：用更强 LLM 评分，注意位置偏差（随机化选项）、自评偏差（交叉验证）、成本
- **Golden Dataset**：人工标注 (输入, 期望输出) 对 = 核心任务 + 回归测试
- **评测框架**：LangSmith（LangChain 生态）→ Braintrust（通用）→ RAGAS（RAG 专用）→ DeepEval（开源 CI/CD）
- **Eval 流水线**：代码变更→自动触发→运行 Eval Suite→对比基线→生成报告→质量门禁
- **避免陷阱**：过度依赖 LLM Judge、评测集 < 50 个 case、只测 happy path、指标单一

## 二、评测维度

| 维度 | 指标 | 说明 |
|------|------|------|
| **任务完成率** | Success Rate | Agent 是否完成任务目标 |
| **工具调用** | Tool Accuracy | 选择工具和参数的正确性 |
| **效率** | Steps, Latency | 完成任务的步骤数/耗时 |
| **回复质量** | Factuality, Relevance | 准确性、相关性、完整性 |
| **安全性** | Guardrail Pass Rate | 是否执行了危险操作 |
| **鲁棒性** | Error Recovery Rate | 面对异常输入的表现 |

## 三、Evals 方法论

### 1. LLM-as-Judge

用更强的 LLM 评审判定 Agent 输出质量：

```yaml
评测 prompt:
  你是一个评测者。请对以下 Agent 输出打分（1-5）：
  - 正确性：答案是否准确
  - 完整性：是否涵盖所有必要信息
  - 工具使用：是否选择了正确的工具

  Agent 输出: {output}
  Ground Truth: {expected}
```

**关键考量**：
- **位置偏差**：LLM 倾向偏好第一个选项 → 随机化选项位置
- **自评偏差**：模型不应对自己输出评分
- **成本**：Judge 模型也消耗 token
- 多次评测取均值、使用不同 Judge 模型交叉验证

### 2. Golden Dataset

人工标注的 (输入, 期望输出) 对。核心任务 + 回归测试。

### 3. A/B 测试

- **离线**：历史数据集上对比
- **在线**：生产环境分流，对比用户满意度

### 4. 红队测试

专门尝试让 Agent 产生有害/错误输出，评估安全边界。

## 四、评测框架

| 平台 | 特点 | 适用 |
|------|------|------|
| **LangSmith** | LangChain 生态，trace + eval | LangChain 用户 |
| **Braintrust** | 独立平台，评测逻辑自定义 | 通用 |
| **RAGAS** | RAG 专用，Faithfulness + Relevance | RAG |
| **DeepEval** | 开源，多种 metric 预设 | CI/CD 集成 |
| **SPIN（自定义）** | 自建 Eval Harness | 高定制需求 |

### 自定义 Eval 核心组成

1. **Test Cases**（测试用例集）
2. **Agent Runner**（运行被测 Agent）
3. **Evaluator**（Judge 模型或规则引擎）
4. **Reporter**（结果汇总和可视化）
5. **CI Integration**（自动触发评测）

## 六、最佳实践

### 设计原则

1. 从核心任务开始，逐步扩展
2. 覆盖常见、边缘、异常场景
3. 定期更新 Evals
4. 固定 seed/temperature 保证可复现
5. 分层评测：工具调用层 → 任务层 → 用户体验层

### 避免的陷阱

| 陷阱 | 说明 |
|------|------|
| 过度依赖 LLM Judge | LLM 有偏差，需结合规则和人工 |
| 评测集过小 | 少于 50 个 case 结果不可靠 |
| 只测 happy path | 必须含异常和边界 |
| 指标单一 | 准确率+效率+安全缺一不可 |
| 基线丢失 | 需保存历史结果做对比 |

## 自测

1. **LLM-as-Judge 有哪些偏差？如何缓解？**
   <br/>→ 位置偏差（倾向第一个选项→随机化位置）、自评偏差（模型对自己输出评分偏高→用不同 Judge 模型交叉验证）、长度偏差（偏好更长回答→控制长度变量）。缓解：多次评测取均值、结合规则检查、人工抽样验证。

2. **为什么评测集不能只测 happy path？至少需要覆盖哪些场景？**
   <br/>→ Happy path 只能验证功能正常，无法暴露鲁棒性问题。还需要覆盖：边界条件（空输入/超长输入）、异常输入（格式错误/非法参数）、工具调用失败场景、安全攻击（Prompt Injection）、并发/超时等压力场景。
