## 一、什么是 AI Agent

> Agent = LLM + 工具 + 记忆 + 规划

Agent 将 LLM 与外部工具、API、数据库连接，使其能执行多步骤任务而非单轮问答。

## 二、Agent 架构模式

### 1. ReAct（Reasoning + Acting）

交替进行**思考**和**行动**：

```
Thought: 需查询数据库获取用户订单
Action:  query_database("SELECT * FROM orders WHERE user_id=123")
Observation: [订单1, 订单2]
Thought: 已获得订单，现在总结回复
Answer: 您有2个订单...
```

最简单高效的模式，大多数框架基于此。

### 2. Plan-and-Execute

先制定完整计划再逐步执行：

```
Plan:
  Step 1: 查询用户订单
  Step 2: 查询物流状态
  Step 3: 计算预计到达时间
  Step 4: 生成回复

Execute: 1 → 2 → 3 → 4
```

计划可在执行中动态调整。Claude Code 接近此模式。

### 3. Multi-Agent 协作

| 模式 | 说明 | 场景 |
|------|------|------|
| Sequential | A → B 流水线 | 串行任务 |
| Hierarchical | Master 分配子任务 | 复杂项目 |
| Debate | 多 Agent 讨论达成共识 | 需要多视角 |
| Parallel | 同时执行不同子任务 | 独立并行 |

**代表框架**：CrewAI、AutoGen、LangGraph

## 三、工具调用（Function Calling）

### 工具定义原则

```json
{
  "name": "search_orders",
  "description": "查询用户订单。返回指定日期范围内的订单列表",
  "parameters": {
    "type": "object",
    "properties": {
      "user_id": { "type": "integer", "description": "用户ID" },
      "start_date": { "type": "string", "description": "起始日期 ISO 格式" }
    },
    "required": ["user_id"]
  }
}
```

### 设计原则

| 原则 | 说明 |
|------|------|
| **描述清晰** | LLM 完全依赖 description 选择工具 |
| **参数简洁** | 减少必填参数，使用合理默认值 |
| **返回结构化** | 明确成功/失败状态 + 数据/错误 |
| **错误可理解** | 错误消息让 LLM 能推理下一步 |
| **幂等安全** | 写操作需确认，支持重试 |

### 调用流程

```
用户输入 → LLM 决策调用哪个工具 → 执行工具 → 结果返回 LLM → 生成回复
                                                ↑___________|
                                                (可能多次调用)
```

### 错误处理

- **重试**：参数错误时 LLM 修正参数重试
- **降级**：工具不可用时告知用户
- **澄清**：信息不足时反问用户
- **超时**：合理超时 + 最大重试次数

## 四、记忆系统

| 层级 | 存储 | 生命周期 | 用途 |
|------|------|:------:|------|
| **工作记忆** | 上下文窗口 | 单次对话 | 当前任务状态 |
| **短期记忆** | 会话状态 | 多轮对话 | 对话历史 |
| **长期记忆** | 向量存储 + DB | 永久 | 知识库、用户画像 |

### 上下文窗口管理

- **滑动窗口**：保留最近 N 轮
- **摘要压缩**：早期对话生成摘要
- **选择性保留**：只保留相关历史
- **结构化状态**：关键信息存为 JSON

## 五、安全防护

### 攻击类型与防护

| 攻击 | 说明 | 防护 |
|------|------|------|
| **Prompt Injection** | 用户输入覆盖系统指令 | 输入验证、指令隔离 |
| **Tool Output Injection** | 工具返回恶意内容 | 输出验证、沙箱 |
| **Data Exfiltration** | Agent 泄露敏感数据 | 输出过滤、最小权限 |

### 核心原则

1. **最小权限**：工具只给必要权限
2. **Human-in-the-Loop**：关键操作需人工确认
3. **沙箱执行**：代码在隔离环境运行
4. **Content Filtering**：输入输出双向过滤

## 六、多 Agent 框架对比

| 框架 | 语言 | 特点 |
|------|:---:|------|
| **LangGraph** | Python/JS | 有状态图编排，精细控制 |
| **CrewAI** | Python | 角色化 Agent，简单易用 |
| **AutoGen** | Python | 微软，对话式多 Agent |
| **OpenAI Agents SDK** | Python | 轻量级，原生工具支持 |
| **Claude API** | — | 原生 tool use + computer use |

### 选型建议

- 简单任务 → LLM SDK 原生 Tool Use
- 有状态流程 → LangGraph
- 多角色协作 → CrewAI / AutoGen
- 生产级 → 自建框架 + 完善评估体系
