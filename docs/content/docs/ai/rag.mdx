## 一、什么是 RAG

**Retrieval Augmented Generation** — 回答问题前先检索相关文档，让 LLM 基于检索到的知识回答。

> RAG = 检索（找到相关内容）+ 增强（拼入上下文）+ 生成（回答问题）

**核心价值**：回答训练外的新知识、减少幻觉、无需频繁微调模型。

## 二、核心流程

```
文档预处理（加载→分块→向量化）
        ↓
   向量存储（Pinecone/Milvus/PGVector/Weaviate）
        ↓
用户问题 → Embedding → 相似度检索 → Top-K 文档 → 拼入 Prompt → LLM 生成
```

## 三、文档分块策略

分块是 RAG 最关键的一步。

| 策略 | 原理 | 适用 |
|------|------|------|
| 固定大小 | 按字符/Token 数 | 通用 |
| 句子分割 | 按句号/换行 | 自然语言 |
| 语义分割 | 模型判断语义边界 | 高质量 |
| 递归分割 | 先段落→句子递归 | 通用推荐 |
| 代码分割 | 按函数/类边界 | 代码库 |

### 关键参数

| 参数 | 建议 |
|------|------|
| chunk_size | 256–1024 tokens |
| chunk_overlap | 10–20%（避免信息在边界割裂） |

## 四、Embedding 模型

| 模型 | 维度 | 特点 |
|------|:---:|------|
| OpenAI text-embedding-3-small | 512/1536 | 性价比 |
| OpenAI text-embedding-3-large | 256/3072 | 精度高 |
| Cohere Embed v3 | 1024 | 多语言 |
| BGE-M3（开源） | 1024 | 中英文，稠密+稀疏 |

## 五、检索策略

| 策略 | 说明 |
|------|------|
| **稠密检索** | Embedding 余弦相似度 |
| **稀疏检索** | BM25 / TF-IDF 关键词 |
| **混合检索** | 稠密 + 稀疏互补 |
| **Multi-Vector** | 多个向量表示不同方面 |

## 六、重排序（Re-ranking）

粗检索 Top 100 → Re-ranker 精排 → 取 Top 5-10 送入 LLM。

| 方案 | 特点 |
|------|------|
| **Cross-Encoder** | 精度最高，较慢 |
| **Cohere Rerank** | API 服务，易用 |
| **BGE-Reranker** | 开源可自部署 |

## 七、高级 RAG 模式

### Self-RAG

LLM 生成过程中自我评估是否需要检索，以及检索结果是否相关。

### Graph RAG

用知识图谱替代纯文本分块。实体关系图 + 社区摘要，适合全局性问题。

代表：Microsoft GraphRAG。

### Agentic RAG

Agent 自主决定何时检索、检索什么、如何组合多次检索结果。支持多步推理和工具组合。

### Parent Document Retrieval

检索较小 chunk（精准匹配），将包含该 chunk 的较大"父文档"送入 LLM（保留上下文）。

### Corrective RAG

检索后先评估相关性，不相关时自动补充检索或回退到网络搜索。

## 八、评估与常见问题

### RAGAS 评估维度

| 维度 | 指标 | 说明 |
|------|------|------|
| 检索质量 | Context Recall, Precision | 检索到的文档是否相关 |
| 答案质量 | Faithfulness, Relevance | 答案是否基于检索内容/是否回答用户问题 |

### 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 答案不完整 | chunk 太小 | 扩大 chunk 或 parent document |
| 检索不相关 | embedding 不匹配 | 微调 / 混合检索 |
| 答案陈旧 | 知识库未更新 | 增量索引 |
| 幻觉 | 检索结果被忽略 | Re-ranking + Self-RAG |
