## 应用场景

| 场景 | 说明 |
|------|------|
| **削峰限流** | 高并发进队列，后端按能力消费 |
| **解耦** | 下游故障不影响上游 |
| **异步处理** | 耗时操作异步化 |
| **数据分发** | 一对多广播（如 Kafka 大数据管道） |

## 架构核心

- **异步解耦模型**：Producer → Broker → Consumer
- **可靠性**：生产确认 → 存储持久化 → 消费确认 → 死信队列
- **吞吐与延迟**：RabbitMQ 低延迟复杂路由；Kafka 海量日志（顺序写+零拷贝）；RocketMQ 兼顾高吞吐与事务消息

## Top 面试问题

### Q1：如何保证消息不丢失？

| 环节 | RabbitMQ | Kafka |
|------|----------|-------|
| 生产端 | publisher confirm | acks=all + 幂等生产 |
| Broker | durable 队列 + 持久化消息 | 多副本 ISR 同步 |
| 消费端 | 手动 ack | 手动 commit offset |

> 死信队列作为保底手段。

### Q2：消息重复消费与顺序

**幂等方案**：数据库唯一键、Redis setnx、消费前查重表 + 业务状态机。

**顺序保证**：

| MQ | 方式 |
|-----|------|
| RabbitMQ | 单队列单消费者天然有序 |
| Kafka | 同一 key → 同一分区 = 分区有序 |
| RocketMQ | MessageQueueSelector |

> 顺序消耗吞吐，业务上常用版本号或状态机规避全局顺序依赖。

### Q3：MQ 实现分布式事务最终一致性

**模式**：本地事务 + 消息表 → 定时任务或 MQ 可靠发送 → 消费幂等。

**RocketMQ 事务消息**：
1. 发送 half 消息（对消费者不可见）
2. 执行本地事务
3. 根据结果 commit 或 rollback
4. 未收到二次确认，Broker 回调反查

**架构考量**：设计"可查可补偿"的事务模型，保证幂等与恢复能力。
