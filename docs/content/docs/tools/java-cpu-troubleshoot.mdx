## 基本思路

1. 查找目标 Java 进程 PID
2. 查找 CPU 占用较高线程 PID
3. 查看对应的线程堆栈

## 速查卡

- **排查三步**：jps -l（找PID）→ top -H -p（找高CPU线程）→ jstack（看堆栈，线程ID转16进制）
- **GC排查**：`jstat -gc <pid> <interval>`，关注FGC次数是否持续增长
- **常见原因**：计算密集/死循环 → 内存泄漏导致频繁GC → 锁竞争导致自旋
- **关键命令**：jps（进程列表）→ jstack（线程堆栈）→ jmap -heap（内存映像）→ jstat -gc（GC统计）

## 排查步骤

### 1. 查找进程 PID

```bash
jps -l
```

### 2. 查找 CPU 占用较高线程

```bash
top -H -p <pid>
```

### 3. 查看线程堆栈

将线程 PID 转为 16 进制后，使用 jstack：

```bash
# 假设线程 PID = 5845 → 0x16d5
jstack <pid> | grep -A 10 0x16d5
```

### 4. GC 查看

```bash
jstat -gc <pid> 1000
```

输出各内存区域使用情况和 GC 次数/耗时。

## 常见原因

1. **计算密集型任务长时间运行** — 检查死循环或大量计算
2. **程序死循环** — 检查 while/for 循环退出条件
3. **内存泄漏导致频繁 GC** — 使用 `jstat -gc` 查看，若 FGC 持续增长则可能存在内存泄漏

## 常用 JVM 排查命令

| 命令 | 用途 |
|------|------|
| `jps -l` | 查看所有 Java 进程 |
| `jstack <pid>` | 查看线程堆栈 |
| `jmap -heap <pid>` | 查看堆内存映像 |
| ``jstat -gc <pid> <interval>`` | 查看 GC 统计 |
| `top -H -p <pid>` | 查看进程内线程 CPU 占用 |

## 自测

1. **`top -H -p <pid>` 显示的线程 ID 为什么需要在 jstack 中转 16 进制？**
   <br/>→ top 显示的是十进制线程 ID（TID），而 jstack 输出的线程信息中 `nid` 字段是 16 进制。`printf '%x\n' <tid>` 转换后，`jstack <pid> | grep -A 10 0x<hex>` 定位到具体线程的堆栈。

2. **CPU 高但 GC 也频繁，先排查哪个？如何区分是业务逻辑还是 GC 导致的 CPU 高？**
   <br/>→ 先用 `jstat -gc` 看 GC 频率和耗时。如果 GC 线程（如 "GC task thread"）CPU 高 → 内存问题（堆太小/内存泄漏）。如果业务线程 CPU 高 → 代码问题（死循环/大量计算）。两者可能关联：内存不够 → 频繁 GC → CPU 高。
